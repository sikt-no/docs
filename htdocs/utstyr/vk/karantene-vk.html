Innledning

I sin enkleste form er det kun ruterne og en server med squid/squidguard som trengs for å realisere en karantenenettsløsning. Ruteren vil sende all web-trafikk (port 80) til serveren som omskriver brukerens opprinnelig web-forespørsel og presenterer en annen webside med informasjon om karantenenettet og om hvorfor brukeren har havnet der. I tillegg trenger man også komponenter som setter en aktuell svitsjeport i et karantenenett/VLAN. Arnold i NAV kan for eksempel brukes for dette. I tillleg kan man der man har tatt ibruk 802.1x bruke "fallback-VLAN/unauth-VLAN" på switchene som sørger for å sette en bruker i karantene-vlan'et dersom han ikke klarer å logge seg inn.

Oppsett på ruter

Skru på globalt

ip wccp web-cache redirect-list 108 group-list 8
Vha. denne kommandoen skrur man på WCCP globalt på ruteren. I tillegg setter vi en redirect-list som definerer hva som IKKE skal WCCP'es og en group-list som definerer hvilke Squid-servere som har lov til å opprette GRE-tunnel mot denne ruteren. Disse access-listene ser f.eks slik ut for ruter i halden: access-list 8 permit 158.39.160.109 access-list 8 deny any access-list 108 remark ** Ting som skal unnslippe WCCP access-list 108 deny ip any host 158.39.162.29 access-list 108 permit ip any any

Skru på for ett vlan

interface Vlan333 ip access-group vlan333in in ip access-group vlan333out out ip wccp web-cache redirect in Her setter vi at all web-trafikk (port 80) inn på VLANet skal redirigeres. På VLANene har vi i tillegg definert èn inngående og èn utgående access-liste. Dette for at i enkelte karantenenett skal det være mulig å gjøre enkelte ting (f.eks lese og sende mail). Slike åpninger defineres da i disse access-listene.

Oppsett på verktøykasse

halden-vk har ett ferdig oppsett med de rette filene i subversion, og kan brukes som mal. Pass på å gjøre lokale tilpasninger som passer per vk ved kopiering.

squid3

Alle verktøykasser er satt opp default til ikke å starte squid. For at verktøykassen skal kjøre squid er du nødt til å legge til ei tom fil i cfengine per vk i katalogen /etc/default/squid. (Fra før finnes det ei fil der som sier exit)

/etc/network/interfaces

Denne fila vedlikeholdes via subversion, og endringer må gjøres der. For at GRE-tunnelen (mellom ruter og server) skal fungere opp mot de forskjellige ruterene, så må det legges inn ett tunnel-interface for hver ruter som ser slik ut. NB: her brukes IP-adressen til loopback interfacen til ruteren som tunnel-endepunkt i motsetning til i squid.conf der vi ikke kan bruke loopback-interfacen sin IP-adresse. 

auto 'tunnellnavn'
    iface 'tunnellnavn' inet static
    address 'adresse for vk'
    netmask 255.255.255.255
    mtu 1476
    pre-up iptunnel add 'tunnellnavn' mode gre local 'adresse for vk' remote 'loopback for ruter'
    post-down iptunnel del 'tunnellnavn'
Her er ett ekspempel ved hiof: 

auto tun-c6500-h-1
    iface tun-c6500-h-1 inet static
    address 158.39.160.109
    netmask 255.255.255.255
    mtu 1476
    pre-up iptunnel add tun-c6500-h-1 mode gre local 158.39.160.109 remote 128.39.103.5
    post-down iptunnel del tun-c6500-h-1
Her er adressen på tunnel-interfacet lik IP-adressen til serveren, men med maske 255.255.255.255. MTU settes til 1476 for å slippe fragmentering av GRE-pakkene. HUSK: Ett slikt interface for HVER ruter som serveren skal kunne snakke med.

/etc/squid/squidGuard.conf

Slik ser squidGuard.conf ut: 


logdir /var/log/squid3

# Her er det kun definert èn klasse
# Du kan lage flere klasser og filtrere på avsenderadresse. Da kan du redirigere til forskjellige sider
src abuse {
        ip 0.0.0.0/0
}

dest aapne {
	# Sier hvilke url/domener som ikke skal redirigeres.
        # Disse ligger under /var/lib/squidguard/db/
        urllist aapne_url
        domainlist aapne_domain
}

# Husk at du her må redirigere til ei webside som ligger i aapne_domain eller aapne_url
acl {
        abuse {
                pass aapne none
                redirect 302:http://karantene.hix.no
        }

        default {
                pass aapne none
                redirect 302:http://karantene.hix.no
        }
}

/proc/sys/net/ipv4/ip_forward

Denne vedlikeholdes via subversion, og det må stå ett 1-tall i fila for at redirigering kan skje via iptables.

IPtables

I filene /etc/network/if-up.d/squid3 og /etc/network/if-down.d/squid3 legger vi inn oppstart/fjerning av iptables-regel når interface går opp eller ned. if-up.d/squid3 ser slik ut: #! /bin/sh -e # Insert iptables rule when tunnell interface is going up if [ "$IFACE" = tun-c6500-h-1 ]; then iptables -t nat -A PREROUTING -i tun-c6500-h-1 -p tcp --dport 80 -j REDIRECT --to-ports 3128 fi I if-down.d/squid3 fjerner vi den samme regelen: #! /bin/sh -e # Remove iptables rule when tunnell interface is going down if [ "$IFACE" = tun-c6500-h-1 ]; then iptables -t nat -D PREROUTING -i tun-c6500-h-1 -p tcp --dport 80 -j REDIRECT --to-ports 3128 fi tun-c6500-h-1 refererer her til tunnellinterfacet i /etc/network/interfaces. Har man flere tuneller må man ha egne innslag for hver tunnell. Dette sørger for at trafikk inn på tunnel-interfacet (fra ruteren) på TCP-port 80 blir REDIRECT'et til TCP-port 3128 hvor squid lytter.
